## Anti-Patterns Verdict

**LLM assessment:** The project is not a first-draft AI generation — it has deliberate design thinking (DESIGN.md, PRODUCT.md, design tokens, font pairing). However, several AI-slop tells have crept in:

- **Glassmorphism creep**: `glass-panel` with `backdrop-filter: blur(12px)` + `border: 1px solid rgba(255,255,255,0.05)` is used as the default wrapper for the case form wizard (`CaseForm.tsx:591`). The DESIGN.md itself mandates glassmorphic panels as the modal/wizard surface. This is explicitly called out as an anti-pattern ("glassmorphism as default") in the impeccable design rules.
- **Emoji as icons**: Dashboard quick links use 📋, ✅, 🎯, 📊 as primary visual identifiers (`DashboardContent.tsx:270-282`). This reads as consumer-app casual, not enterprise clinical.
- **Nested cards**: Goals page wraps HeroUI `<Card>` with `className="panel"` creating double-bordered containers (`goals/page.tsx:70`). Reports page does the same. The DESIGN.md anti-pattern list says "No nested hard-bordered cards" — and then the code does exactly this.
- **Identical card grids**: The 4 KPI rings are structurally identical components in a 2x2 grid, and the 4 quick-link cards are identical panels with emojis. While appropriate for KPI rings, the quick links are the "icon + heading + text" card grid pattern.
- **Top-stripe borders**: Reports page uses `border-t-3 border-t-success` etc. on metric cards — a variation on the banned side-stripe pattern (`reports/page.tsx:95-112`).

**Deterministic scan:** The automated detector returned zero findings (clean scan). The detector catches specific CSS/HTML anti-patterns (gradient text, sketchy SVGs, excessive border-radius, etc.) that this project largely avoids.

**Visual overlays:** No browser injection available in this session (OpenCode CLI). Skipping live overlay.

## Overall Impression

E-Logbook has a strong foundation: the dark clinical palette is distinctive, the font triad (Outfit/Inter/Geist Mono) is well-chosen, and the de-identification toggle is genuinely smart domain UX. But the project undermines itself with junior mistakes — emojis where SVG icons belong, glass panels as a default surface, error messages that leak database internals, and a complete absence of loading/empty state design. It feels like a v0.7 that needs a focused quality pass to reach the "premium, fluid, and advanced" standard its own PRODUCT.md promises.

**Biggest opportunity:** Replace decorative elements (emojis, glass panels, nested cards) with clinical-precision alternatives, add skeleton states and contextual feedback, and sanitize error surfaces.

## What's Working

1. **De-identification toggle** (`CaseForm.tsx:321-330`): Switches between full PII and HIPAA-safe mode with clear warnings per mode. The dual-path form (MRN+DOB vs Age+Hash) with a single toggle is excellent domain-specific UX. Most clinical tools bury this in settings.

2. **Color-meaningful status badging**: Draft → neutral, Pending → amber glow, Approved → emerald glow, Rejected → crimson glow. Each badge has a distinct visual character (not just a color swap). The `box-shadow: 0 0 6px` glow is restrained enough to not be the banned "ghost-card" pattern.

3. **Role-filtered navigation** (`[tenant]/layout.tsx:49`): Sidebar links are filtered by user role. A resident never sees Admin or Audit. Reduces cognitive load and prevents dead-end navigation. This is good product thinking.

## Priority Issues

### [P0] No active navigation state in sidebar
**Why it matters:** Users cannot tell which page they're currently on. In a 8-section clinical tool, this is disorienting — especially when switching between Cases, Approvals, and Reports rapidly. Every medical resident and supervisor will feel lost.

**Fix:** Add an `active` state to sidebar links (background highlight, text color change, left accent bar). Use Next.js `usePathname()` to detect the current route.

**Suggested command:** `$impeccable layout`

### [P0] Raw database errors exposed in UI
**Why it matters:** `error.message` from Supabase is displayed directly to users (`CaseForm.tsx:90`, `ApprovalsDashboard.tsx:71`, `cases/page.tsx:47`). These can expose table names, column names, and query structure. For a HIPAA-context clinical app shipping audit and compliance features, leaking schema details is a genuine privacy concern — even if no PHI is in the error, it reveals data model internals.

**Fix:** Wrap all error display in a sanitizer that maps known error codes to user-facing messages. Never render `error.message` directly.

**Suggested command:** `$impeccable harden`

### [P1] Glassmorphism as default wizard surface
**Why it matters:** `glass-panel` with backdrop blur is the outer wrapper for the entire case form (`CaseForm.tsx:591`). Glass effects are meant for transient overlays (modals, dropdowns), not persistent form containers. The semi-transparent background reduces text contrast against the dark backdrop, making form labels and values harder to read. It also reads as a 2023-era design trend.

**Fix:** Replace `glass-panel` on the CaseForm wrapper with `panel`. Reserve `glass-panel` only for actual overlays/modal contexts.

**Suggested command:** `$impeccable quieter`

### [P1] No skeleton loading states
**Why it matters:** Throughout the app, loading is communicated with plain `"Loading..."` text (`DashboardContent` depends on server data but has no loading state component, `ApprovalsDashboard.tsx:90`, `CaseForm.tsx:289` uses `isLoading` on Select). The product register explicitly requires skeleton states. Plain text loading causes layout shift and feels unpolished.

**Fix:** Add skeleton components (pulsing rectangles matching content shape) for the dashboard KPI rings, recent cases panel, goals panel, and case list table. HeroUI likely provides skeleton primitives.

**Suggested command:** `$impeccable polish`

### [P1] Emojis in navigation and quick links
**Why it matters:** `DashboardContent.tsx:270-282` uses 📋, ✅, 🎯, 📊 as the primary visual identifiers for navigation cards. These render inconsistently across platforms and read as consumer-grade, not clinical-enterprise. A tool promising "premium, fluid, and advanced" UI should not use emoji as its icon system.

**Fix:** Replace with consistent SVG icons (HeroUI icon set or a clinical icon library like Phosphor). Use the same icon style across sidebar nav and quick links.

**Suggested command:** `$impeccable bolder`

### [P2] Inconsistent status display vocabulary
**Why it matters:** The dashboard renders status with custom glowing badges (`StatusBadge` component with `badge-draft`, `badge-pending`, etc.) while the cases table uses HeroUI `<Chip>` components. Two different visual languages for the same concept. The badges have glowing shadows; the chips are flat. When a user moves from dashboard to cases list, the statuses look like they belong to different products.

**Fix:** Standardize on one status component. Either use HeroUI Chip everywhere (and remove custom badge CSS) or use custom badges everywhere (and remove Chip from the table).

**Suggested command:** `$impeccable polish`

### [P2] Nested card patterns
**Why it matters:** Goals and Reports pages wrap HeroUI `<Card>` components with `className="panel"` — this creates a card inside a card (HeroUI has its own border/shadow, plus the panel class adds another). DESIGM.md explicitly bans this. The visual result is double-bordered containers that add visual noise without purpose.

**Fix:** Use either HeroUI Card (and style it via its own props) or a plain `<div className="panel">`. Never both.

**Suggested command:** `$impeccable distill`

## Persona Red Flags

### Alex (Impatient Power User — Supervisor/Attending)
- **No keyboard shortcuts**: Can't press `A` to approve, `R` to reject. Must click through each pending case individually.
- **No bulk actions**: 15 pending cases requires 15 separate approval interactions, each with a scroll to find the comment field and button.
- **No search/filter on cases list**: Must visually scan the table to find specific cases. No MRN search, no date range filter.
- **Loading text instead of skeletons**: "Loading..." text causes layout shift, forcing re-orientation when content arrives.
- **Forced wizard steps**: Can't jump directly to Case Details from Template selection. Must click through Patient Info even when fields are pre-filled.

### Jordan (Confused First-Timer — New Medical Resident)
- **Template dropdown is a flat list with no guidance**: 20+ templates shown as `"Specialty - Name"` strings with no categorization, search, or "commonly used" section. Jordan will scroll and guess.
- **No tooltips or inline help**: "Patient MRN (encoded client-side)" is technical — Jordan may not understand what base64 encoding means or why it matters.
- **No confirmation after case submission**: Redirect to cases list without a success message. Jordan will wonder "Did it save?" and navigate back to check.
- **No empty state guidance**: "No cases logged yet" with a tiny "Click Log New Case to get started" is the only guidance. No sample case, no template preview, no "what to expect" copy.

### Sam (Accessibility-Dependent User)
- **Color-only status indicators**: The status distribution on the Reports page uses 4 colored circles with labels — but if someone can't distinguish amber from emerald, the chart is meaningless without additional shape or pattern differentiation.
- **Custom SVG KPI rings lack ARIA labels**: The animated SVG circles on the dashboard have no `role="progressbar"`, no `aria-valuenow`, no `aria-valuemin/max`. Screen readers will skip them entirely.
- **Emoji-only quick links**: 📋 Cases, ✅ Approvals — emojis have unpredictable screen reader announcements (varies by OS and reader). Some read "clipboard", some read "clipboard emoji", some skip.
- **Focus indicators may not be visible on dark theme**: The project uses HeroUI's default focus rings, but custom-styled inputs (login page, `className="focus:ring-1 focus:ring-primary-glow"`) may not provide sufficient contrast on the `#060814` backdrop.

## Minor Observations

- Login page uses a raw `<input>` while the rest of the app uses HeroUI `TextField` — inconsistent form vocabulary.
- "De-identified" status is shown with the `badge-approved` class (green) on the approvals dashboard — this is semantically wrong; de-identification is a privacy mode, not an approval status. It should use its own neutral badge variant.
- The review step in CaseForm shows "MRN Hash" with a long base64 string truncated at 200px — on smaller screens this is unreadable and offers no value to the user.
- `CaseForm.tsx:595` uses `bg-danger-50 text-danger` but the project's theme is dark. HeroUI's `danger-50` is likely a light-mode color. This error banner will probably render as near-white on dark.
- Reports page has a `border-t-3` class — Tailwind v4 requires explicit border width configuration. If this CSS is untested it may silently fail.
- The mobile bottom nav has no active state indicator (same P0 issue as desktop sidebar).
- Approval buttons lack a confirmation step — clicking "Reject" immediately rejects a resident's case with no "Are you sure?" dialog. This is a high-stakes action.

## Questions to Consider

- What if the case form wizard were a single scrollable page with anchored sections instead of 4 discrete steps? Power users could fill it in 20 seconds.
- What if "Log New Case" were a floating action button always accessible from any page, not just the dashboard header?
- Does the approval queue need to be a full page, or could it be a slide-out panel overlaid on the cases list so supervisors stay in context?
- What would this interface look like if it only used the Outfit + Geist Mono font pair (dropping Inter for a single-sans approach, as the product register suggests)?
